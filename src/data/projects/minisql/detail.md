## 项目简介

MiniSQL 是数据库系统课程中的综合型项目，目标是从底层开始实现一个精简的单用户 SQL 引擎。系统需要支持基本的建表、建索引、插入、删除、更新、查询等操作，并通过索引提升查询效率。

项目仓库：[database-system_-cl_2022_-spring](https://gitee.com/WhiteScale/database-system_-cl_2022_-spring.git)

## 支持能力

MiniSQL 的核心需求主要包括：

- 支持 `integer`、`float`、`char(n)` 三类基础数据类型
- 支持表创建、索引创建与删除
- 支持主键和 `unique` 约束
- 支持 `insert / delete / update / select`
- 支持通过 `and / or` 组合多个查询条件
- 支持等值查询与区间查询
- 为主键和部分唯一属性建立 B+ 树索引

从作品集角度看，这个项目不只是“写一个 SQL 解释器”，而是把一个小型数据库从存储层、索引层、元数据管理层到执行层串起来。

## 系统架构

根据文档，整个 MiniSQL 可以分成几个核心模块：

### 1. Disk Manager 与 Buffer Pool Manager

这一层位于系统底部，负责数据库文件中的数据页分配、回收、读写，以及页缓冲管理。

- Disk Manager 负责磁盘页的定位和读写
- Buffer Pool Manager 负责把页缓存到内存中
- 缓冲池需要支持页面替换、脏页刷新和页状态维护

这一部分让我更直观地理解了数据库为什么不是“直接从文件读记录”，而是围绕 **页（page）** 作为基本 I/O 单位来组织整个系统。

### 2. Record Manager

Record Manager 负责表中记录的组织与访问。文档中采用的是 **堆表（Table Heap）** 形式：

- 记录以无序方式存储
- 一个表由多个数据页组成
- 支持记录插入、删除和查找
- 记录、模式、列、域等抽象都被拆成独立的数据结构

这一层更接近“真正的数据存储”，让我对一条 SQL 记录最终如何落在磁盘页里有了更具体的认识。

### 3. Index Manager

索引部分基于 **B+ 树** 实现。文档中明确要求支持：

- 索引创建和删除
- 等值查找
- 范围查找
- 键值插入与删除

同时，B+ 树结点与数据页大小保持一致，这也让索引结构和底层页管理天然衔接起来。  
从实现角度看，这部分是整个项目里很核心的一环，因为它直接决定了查询效率优化是否真正成立。

### 4. Catalog Manager

Catalog Manager 负责维护数据库的模式信息，包括：

- 表的定义信息
- 列的定义信息
- 索引的定义信息

除了在内存中维护这些元数据外，还需要支持持久化。也就是说，数据库重启之后，表和索引的定义不能丢失。  
这一模块让我更深地理解了“数据库不仅要存数据，还要存数据的定义”。

### 5. SQL Executor 与 Parser

执行层负责把 SQL 语句真正落到 Record Manager、Index Manager 和 Catalog Manager 上。

文档中给出的整体流程是：

1. Parser 对输入 SQL 做词法和语法分析
2. 生成语法树
3. Executor 根据语法树分发到对应执行函数
4. 调用各底层模块完成执行
5. 把执行结果返回给上层

这一部分对应的是数据库系统里非常典型的一条链路：  
**SQL → 语法树 → 执行逻辑 → 底层存储/索引接口**

## 典型实现内容

- 建表时解析字段定义、主键和唯一约束
- 在建索引时维护表和索引的元信息关系
- 在插入记录时同步更新索引
- 在查询时根据条件选择遍历记录或使用索引
- 在删除、更新时同步维护堆表与索引结构

如果从数据库课程学习的角度看，这类项目最大的价值在于：  
它把“表、记录、页、索引、元数据、执行器”这些原本分散的概念真正放到了一个统一系统里。

## 项目收获

这个项目帮助我把数据库系统课程里的很多概念连成了完整链路：

- 从磁盘页和缓冲池理解底层存储
- 从堆表理解记录组织方式
- 从 B+ 树理解索引如何优化查询
- 从 Catalog Manager 理解元数据管理
- 从 Parser / Executor 理解 SQL 是如何被解释和执行的

相比只做单一算法题或局部实验，MiniSQL 更像一个“小型数据库内核”的缩影。对我来说，它最大的收获是把数据库系统从抽象概念变成了可以动手实现、调试和验证的工程对象。
