## 项目简介

该项目是编译原理课程大程，小组实现了一门类 C 语言 **C Reduced** 的完整编译流程：源程序先经过词法分析、语法分析和语义分析，再生成中间代码，最后由解释器执行。

项目仓库：[minicompiler](https://github.com/Ceceliawai/minicompiler.git)

## 语言特性

根据实验报告，C Reduced 支持了较完整的基础语言能力，包括：

- 基本类型：`int`、`float`、`char`、`bool`、`void`
- 指针与结构体：支持 `int*`、`char*`、`float*`、`bool*` 与 `struct`
- 控制流：`if / else / for / while / break / continue / return`
- 表达式：算术、取模、逻辑、赋值、`sizeof`、函数调用、数组访问、结构体域访问
- 数组：支持多维数组与可变长度数组，如 `int a[m][n]`
- 结构体：支持结构体定义、结构体域访问，以及同域结构体之间的直接赋值
- 输入输出：通过特别定义的 `scanf` 和 `printf` 完成基础 I/O

## 实现流程

整个系统的执行链路为：

1. 对 `.cmm` 源程序进行词法分析与语法分析
2. 在抽象语法树基础上进行语义检查
3. 若无错误则生成 `.ir` 中间代码
4. 使用解释器读取中间代码并从 `main` 开始执行

报告中给出的实现环境主要为 **C++11 + Flex + Bison**，并在课程项目中完成了从前端到执行端的一体化实现。

## 关键实现

### 1. 前端分析

- 使用 **Flex** 完成词法分析
- 使用 **Bison** 描述文法并完成语法分析
- 构建语法树节点 `Node`，为后续语义分析和中间代码生成提供统一结构

### 2. 语义分析

语义分析阶段重点检查：

- 变量、函数、结构体是否已定义
- 函数声明与定义是否一致
- 返回值类型是否匹配
- 数组下标是否合法且是否为整数
- 结构体域访问是否合法
- 类型转换与赋值是否符合约束

### 3. 中间代码生成

项目没有直接生成机器码，而是先生成自定义 IR。这样做的好处是可以先把前端分析和执行部分解耦：前端只负责把源程序翻译成统一的中间表示，后端再针对该表示完成执行。

从报告中的样例可以看到，中间代码已经能够表达：

- 函数定义与参数传递
- 条件跳转与标签跳转
- 算术运算与赋值
- 数组访问
- 返回值传递

例如快速排序样例中，会出现 `FUNCTION`、`PARAM`、`IF ... GOTO LABEL`、`RETURN` 这类指令，说明项目已经把结构化语句转换成了更适合执行的线性 IR。

#### 变量重命名与作用域

中间代码里一个很关键的点是 **变量重命名**。  
因为源语言允许不同作用域里出现同名变量，但在中间代码和解释器执行阶段，变量更接近“统一符号表中的名字”。如果不做重命名，不同作用域中的同名变量就很容易互相覆盖。

因此，这个项目在中间代码生成测试中专门覆盖了“变量重命名”场景。可以把它理解为：

- 源程序里同名但不同作用域的变量
- 在 IR 中会被改写成不同名字
- 从而把“作用域”信息显式编码进变量名

例如可以抽象理解成下面这种形式：

```text
源程序：
int i;
{
  int i;
}

IR（示意）：
INT i1
INT i2
```

这种做法的好处是：

- 避免不同作用域中的同名变量冲突
- 降低解释器执行时的作用域维护复杂度
- 让赋值、跳转和函数调用时引用的变量更明确

也正因为如此，报告中的中间代码生成测试会特别关注“多函数、有循环、有条件分支、有重命名”的组合场景。

### 4. 解释器

解释器部分负责读取 `.ir` 文件并真正执行中间代码。报告里给出的实现思路比较清晰，核心是 **两遍扫描**：

- **第一遍扫描**：建立函数与标签符号表
- **第二遍扫描**：从 `main` 开始顺序执行中间代码

#### 解释器的核心数据结构

为支持函数调用、跳转和运行时状态恢复，解释器中设计了几类关键结构：

- 符号表 `symbolTable`：维护函数、标签和变量
- 程序计数器 `PC`：指向当前执行到的中间代码行
- `Program_`：保存完整 IR 程序内容
- 函数状态 `FunctionState`：记录调用前的上下文
- 函数调用栈：支持嵌套调用与递归返回

#### 解释器怎么执行

根据报告，解释器执行过程大致可以概括为：

1. 读入中间代码
2. 第一轮扫描时记录所有函数入口和标签位置
3. 第二轮从 `main` 开始执行
4. 每读取一行，根据首个 token 判断这是赋值、跳转、函数调用还是返回
5. 遇到 `GOTO` 就修改 `PC`
6. 遇到函数调用就保存当前状态、压栈、跳转到被调函数
7. 遇到 `return` 就弹栈并恢复到调用前状态

也就是说，解释器本质上是在用 C++ 手动模拟一个“简化版运行时”。

#### 函数调用如何处理

报告里对函数调用的描述比较详细，主要步骤包括：

- 深拷贝当前函数中已定义变量的状态
- 保存当前函数名与 `PC`
- 记录是否有返回值以及返回值写回位置
- 将这些信息封装成 `FunctionState` 压入调用栈
- 把实参依次传给形参
- 把 `PC` 跳转到目标函数入口

而在函数返回时，则会：

- 取出返回值
- 从调用栈中弹出上一层函数状态
- 恢复变量、函数名和调用前的 `PC`
- 如果需要，再把返回值写回对应变量

这一设计很好地解决了递归和嵌套调用中的状态恢复问题。

#### 数组与指针的处理

报告中还特别实现了“多维下标映射到一维存储”的辅助逻辑，用于统一数组和指针的内部表示。  
也就是说，不论源程序里是 `a[3][4]` 还是更高维的访问，解释器最终都会把它换算成底层的一维位置再读取或写入。

这部分设计让我觉得很有代表性：它既体现了编译/解释系统里“高层语义到底层表示”的转化，也说明这个项目不只是停留在前端分析，而是真正把执行链路做通了。

## 测试与结果

报告中既包含官方样例测试，也包含大量自行设计的测试：

- 端到端样例：如快速排序程序，能够完成从源程序到 IR，再到解释执行的完整流程
- 语法测试：覆盖多维数组、错误赋值、非法符号、结构体定义等情况
- 语义测试：覆盖函数声明/定义一致性、结构体赋值、返回值类型、数组访问、域访问等情况
- 中间代码生成测试：覆盖单函数、多函数、循环、条件分支和变量重命名等场景

从实验结果看，项目已经能够正确处理较复杂的课程样例与自定义测试用例，验证了编译前端与解释执行链路的完整性。

## 项目总结

这个项目让我系统地把编译器的几个核心阶段真正串联起来：从词法与语法规则设计，到 AST 与语义检查，再到 IR 生成和解释执行。相比单独实现某一个模块，这类完整链路项目更能帮助我理解“源程序如何一步步被翻译并运行起来”。
