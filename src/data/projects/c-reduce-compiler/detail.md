## 项目简介

该项目是编译原理课程大程，小组实现了一门类 C 语言 **C Reduced** 的完整编译流程：源程序先经过词法分析、语法分析和语义分析，再生成中间代码，最后由解释器执行。

项目仓库：[minicompiler](https://github.com/Ceceliawai/minicompiler.git)

## 语言特性

根据实验报告，C Reduced 支持了较完整的基础语言能力，包括：

- 基本类型：`int`、`float`、`char`、`bool`、`void`
- 指针与结构体：支持 `int*`、`char*`、`float*`、`bool*` 与 `struct`
- 控制流：`if / else / for / while / break / continue / return`
- 表达式：算术、取模、逻辑、赋值、`sizeof`、函数调用、数组访问、结构体域访问
- 数组：支持多维数组与可变长度数组，如 `int a[m][n]`
- 结构体：支持结构体定义、结构体域访问，以及同域结构体之间的直接赋值
- 输入输出：通过特别定义的 `scanf` 和 `printf` 完成基础 I/O

## 实现流程

整个系统的执行链路为：

1. 对 `.cmm` 源程序进行词法分析与语法分析
2. 在抽象语法树基础上进行语义检查
3. 若无错误则生成 `.ir` 中间代码
4. 使用解释器读取中间代码并从 `main` 开始执行

报告中给出的实现环境主要为 **C++11 + Flex + Bison**，并在课程项目中完成了从前端到执行端的一体化实现。

## 关键实现

### 1. 前端分析

- 使用 **Flex** 完成词法分析
- 使用 **Bison** 描述文法并完成语法分析
- 构建语法树节点 `Node`，为后续语义分析和中间代码生成提供统一结构

### 2. 语义分析

语义分析阶段重点检查：

- 变量、函数、结构体是否已定义
- 函数声明与定义是否一致
- 返回值类型是否匹配
- 数组下标是否合法且是否为整数
- 结构体域访问是否合法
- 类型转换与赋值是否符合约束

### 3. 中间代码与解释器

项目没有直接生成机器码，而是先生成自定义 IR，再由解释器执行。报告中提到解释器采用了较清晰的两遍执行策略：

- **第一遍扫描**：建立函数与标签符号表
- **第二遍扫描**：从 `main` 开始执行中间代码

为支持函数调用、返回值传递和递归，解释器中还设计了：

- 符号表 `symbolTable`
- 程序计数器 `PC`
- 函数状态 `FunctionState`
- 函数调用栈

此外，为了处理多维数组与指针访问，项目还实现了“多维下标映射到一维存储”的辅助逻辑，用于统一数组和指针的内部表示。

## 测试与结果

报告中既包含官方样例测试，也包含大量自行设计的测试：

- 端到端样例：如快速排序程序，能够完成从源程序到 IR，再到解释执行的完整流程
- 语法测试：覆盖多维数组、错误赋值、非法符号、结构体定义等情况
- 语义测试：覆盖函数声明/定义一致性、结构体赋值、返回值类型、数组访问、域访问等情况
- 中间代码生成测试：覆盖单函数、多函数、循环、条件分支和变量重命名等场景

从实验结果看，项目已经能够正确处理较复杂的课程样例与自定义测试用例，验证了编译前端与解释执行链路的完整性。

## 项目总结

这个项目让我系统地把编译器的几个核心阶段真正串联起来：从词法与语法规则设计，到 AST 与语义检查，再到 IR 生成和解释执行。相比单独实现某一个模块，这类完整链路项目更能帮助我理解“源程序如何一步步被翻译并运行起来”。
